AIコーディングエージェントのWindsurfとClaude Code、両方を3ヶ月くらいガッツリ使った高校生の率直なレビューです。「結局どっちがいいの?」とよく聞かれるので、料金・実コマンド・つまずいた点までまとめました。

## 形態の違いを最初に押さえる

| 項目 | Windsurf | Claude Code |
|---|---|---|
| 形態 | VS Codeベースの**GUIエディタ** | **CLI**のエージェント |
| 起動 | アプリを開く | `claude` をターミナルで叩く |
| 編集承認 | 差分UIで一つずつクリック | コミット/diffで確認 |
| 設定ファイル | `.windsurfrules` | `CLAUDE.md` |
| 長文コンテキスト | 中〜長(モデル依存) | Sonnet 4.6で**最大1Mトークン** |

僕がClaude Codeに寄せた最大の理由は1Mトークンです。中規模リポジトリを丸ごと食わせて「この機能、責務分割しといて」が通るのは初めて触った時に驚きました。

## 料金感(2026年5月時点)

サブスクと従量の混在で比較が面倒なので実測ベースで。

- **Windsurf Pro**: 月$15前後。フローアクション(エージェントの1ステップ)に上限あり
- **Claude Pro**: 月$20。Claude Codeも同枠で使える。ヘビーに回すと5時間ごとのレートに当たる
- **Claude Max**: 月$100 or $200。僕は$200($200/月 ≒ ¥21,400)を契約中。1日8時間回しても枯れない

Windsurfは上限到達後に「ここから個別課金」になりがちで、月額の見通しはClaudeのMaxの方が楽でした。学校の課題レベルならWindsurf無料枠で十分です。

## 実装例: Claude Codeのプロジェクト設定

CLIで一番効くのが`CLAUDE.md`によるルール固定です。僕は副業のThreads投稿生成スクリプトを書く時、ルートにこれを置いています。

```markdown
# プロジェクト規約
- Pythonは3.12, 型注釈は必須
- HTTPはhttpx(同期), リトライはtenacityで指数バックオフ
- secretsは.envから読み, 平文書き出し禁止
- 投稿文の最終出力は200字以内, 1行目に空行

# テスト
- `pytest -q` が通ること
- 新規関数には最低1ケース追加
```

これを置いた状態で

```bash
claude "fetch_threads.pyのHTTPリトライをtenacityに置き換え, テストも更新"
```

と投げると、リトライ実装→`test_fetch_threads.py`に1ケース追加→`pytest -q`まで自走します。1Mコンテキストだと既存14ファイル全部読んでから動くので、命名規則のズレが出ないのが地味に強い。

## Windsurfのほうが勝つ場面

CSSとReactコンポーネントの調整は完全にWindsurfです。理由は差分プレビュー。Tailwindの`gap-3`を`gap-4`に変える時、ターミナルで`+gap-4 -gap-3`を見るより、エディタのインライン差分とブラウザを並べた方が圧倒的に速いです。

実測で、Threads API Appの管理画面UIを直した時:

- Windsurf: 1コンポーネント約4分(プレビュー見ながら微調整)
- Claude Code: 1コンポーネント約7分(`npm run dev`を別窓で起動→目視→指示し直し)

逆にバックエンドのリファクタは:

- Windsurf: 47ファイル横断で途中で文脈が切れた
- Claude Code: 1Mコンテキストで一発、所要9分

## ハマったポイント

**1. WindsurfのフローアクションをBashで使い切る事故**

CIスクリプトをエージェントに任せたら、エージェントが`npm install`→失敗→再試行を勝手にループしてフローアクションを1時間で枯らしました。**長時間バッチ系はCLIに寄せた方が安全**です。

**2. Claude Codeの権限設定**

最初`Bash(rm:*)`まで自動許可にしてたら、生成ファイルを掃除する指示で`node_modules`まで消えかけました。今は`.claude/settings.json`で

```json
{
  "permissions": {
    "allow": ["Bash(npm test)", "Bash(pytest:*)", "Edit", "Write"],
    "deny": ["Bash(rm -rf*)", "Bash(git push --force*)"]
  }
}
```

と明示。**denyを先に書く**のが事故防止の鉄則だと学びました。

**3. 両方使うと文脈がズレる**

同じファイルをWindsurfとClaude Codeで交互に編集すると、片方のキャッシュが古くて差分がぶつかります。**1リポジトリ1ツールに固定**して、別目的(UI vs バックエンド)で分けるのが現実解。

## 使い分けの結論

3ヶ月の運用で落ち着いた分担はこうなりました。

- **UI調整・小さなバグ修正・学習用の写経** → Windsurf
- **リポジトリ全体のリファクタ・長尺ドキュメント生成・自動化バッチ** → Claude Code
- **CIに組み込む / cron で回す** → Claude Code一択(CLIなので)

迷ったらWindsurf無料枠から触って、コンテキスト長や自動化が物足りなくなったタイミングでClaude Codeを足す順番がコスパいいです。逆にターミナルが苦じゃない人は最初からClaude Code Proで始めて、UI調整の時だけWindsurf無料を併用、で十分回ります。

「画面で見て安心したい」のか「コマンドで殴りたい」のか、自分の手癖で選ぶのが結局一番失敗しないと思います。
